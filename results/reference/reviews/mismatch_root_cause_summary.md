Source run: `sql_rag_vs_pandasai_20260608T043329Z`

This review uses the rechecked comparison where table row ordering is evaluated separately. Row-order-only table mismatches are not counted as semantic failures in the adjusted accuracy.

## Executive Summary

- Adjusted SQL-RAG accuracy is `24/50 = 48%`, up from the strict ordered score of `18/50 = 36%`.
- Table row ordering explains `6` table failures: `table-Q3`, `table-Q6`, `table-Q7`, `table-Q8`, `table-Q14`, and `table-Q16`.
- After ignoring table row order, there are `26` remaining mismatches: `13` value-output and `13` table-output.
- Most reference SQL pairs look logically correct under the benchmark's implicit metric definitions.
- The main source of bad SQL-RAG accuracy is not syntax or execution failure. It is semantic mismatch: the agent repeatedly chose a different business definition from the reference SQL.
- The biggest recurring semantic split is `order-item net revenue = quantity * unit_price - discount_amount` versus `payments - refunds`.
- Several apparent failures are only formatting or output-contract issues: percentages returned as decimals, month labels as timestamps instead of `YYYY-MM`, missing expected columns, unrounded numeric output, or row-limit truncation.

## Reference SQL Correctness Review

The reference SQL was generated by Claude, so I reviewed whether each question/reference pair makes logical sense.

- Overall, the reference SQL suite is coherent and internally consistent.
- The suite consistently defines net revenue from order items as `quantity * unit_price - discount_amount`.
- Delivered metrics consistently filter `orders.order_status = 'delivered'`.
- Promo metrics correctly use promo-code fields `promo_id` and `promo_id_2`, not discount amount.
- Repeat-customer and cohort metrics are generally computed at the correct customer/order grain.
- Table reference SQLs generally include explicit ordering, so strict row order is an expected part of those table outputs unless the evaluator intentionally ignores row order.

Questionable or ambiguous reference definitions:

- `value-Q7`: "return rate" is defined as the share of orders with `order_status = 'returned'`. A different but plausible definition is "orders with rows in the returns table." SQL-RAG chose the latter.
- `value-Q29`: "returned orders" is also defined by `order_status = 'returned'`, while SQL-RAG used the returns table.
- `table-Q10`: `avg_unit_price` is defined as simple `AVG(unit_price)` for delivered item rows. The question could be interpreted as net revenue per delivered unit, but the reference interpretation is reasonable.
- `table-Q18`: zip-code loyalty is grouped by the order/shipping zip from `orders`, not the customer profile zip. This is plausible but should be stated in a benchmark dictionary.

My conclusion: the reference SQL is mostly usable as ground truth, but the benchmark needs a metric dictionary so agents do not have to infer definitions from wording alone.

## Value-Output Mismatches

### value-Q7: Highest return-rate payment method

- Question: Which payment method has the highest return rate, and what is that rate?
- Reference SQL definition: group by `orders.payment_method`; return rate is `100 * AVG(order_status = 'returned')`.
- SQL-RAG definition: join `payments`, `orders`, `order_items`, and `returns`; return rate is share of orders with a returns-table row.
- Ground truth: `payment_method = cod`, `return_rate_pct = 8.9159`.
- SQL-RAG output: `payment_method = cod`, `return_rate = 0.0890247308`.
- Difference: same winning method, but SQL-RAG used the returns table and returned decimal fraction instead of percent.
- Likely cause: ambiguous "return rate" plus no enforced metric definition for returned-order status versus returns-table records.

### value-Q8: Promo revenue share

- Question: What share of total net revenue comes from orders that used at least one promo code?
- Reference SQL definition: promo order if `promo_id` or `promo_id_2` is not `NO_PROMO`; revenue is order-item net revenue.
- SQL-RAG definition: promo order if `discount_amount > 0`; revenue subtracts returns.
- Ground truth: `promo_revenue_share_pct = 29.9121`.
- SQL-RAG output: `promo_share = 0.2980980487`.
- Difference: SQL-RAG used discount amount as promo proxy, subtracted refunds, and returned a fraction.
- Likely cause: semantic context does not teach promo-code fields strongly enough, and output percent normalization is weak.

## Why Accuracy Was So Poor

1. The benchmark uses a hidden metric dictionary, but the agent only gets loose semantic context.

Many questions rely on definitions that are not obvious from natural language. Examples: "net revenue" means item revenue after discounts, not payments minus refunds; "promo" means promo-code columns, not nonzero discounts; "returned" means `order_status = 'returned'` in some questions, not rows in `returns`.

2. SQL-RAG overuses payments and refunds.

The agent repeatedly treats "net revenue" as `payments - refunds`. The reference suite usually treats net revenue as `quantity * unit_price - discount_amount`. This one mismatch family explains many value and table failures.

3. The retrieval documents are too thin for benchmark semantics.

The current `ecommerce_semantics.md` explains order status and basic return logic, but it does not define all benchmark metrics: delivered net revenue, promo flag, repeat-customer denominator, top-decile selection, unit buckets, exact output aliases, categorical casing, and date formatting.

4. Table-output contracts are not enforced.

The agent often gets the calculation close but misses required aliases, required columns, month formatting, percent scaling, row ordering, or row count. Evaluation expects data-frame equality, not just a plausible answer.

5. The execution layer caps table outputs at 100 rows.

Questions like signup cohorts and monthly series need more than 100 rows, but `execute_sql` fetches at most `DEFAULT_ROW_LIMIT = 100`. This creates false shape mismatches even when the SQL logic is otherwise close.

6. Categorical value normalization is underpowered.

The benchmark uses lowercase CSV values such as `female`, while the PostgreSQL core table can expose title-cased values such as `Female`. Without indexed low-cardinality values for `gender`, `payment_method`, `device_type`, `order_source`, and promo fields, the agent guesses.

7. The graph is still Phase 2, not a benchmark-grade multi-candidate/ranking system.

The pipeline generates one candidate and only repairs execution failures. It does not generate competing metric interpretations or rank candidates against benchmark-style rubric criteria.

## Recommended Fix Direction

- Add benchmark metric definitions to `docs/schema_context`, especially revenue, delivered revenue, promo flags, returned orders, repeat-customer rate, date/month formatting, and unit buckets.
- Teach SQL generation to preserve requested output columns exactly, including aliases and percent scale.
- Add table-output mode with a higher row limit or a query-specific row limit.
- Add low-cardinality value retrieval for all categorical dimensions, not only a small hardcoded subset.
- Add Phase 3 multi-candidate generation where candidates explicitly vary metric definitions and are ranked against retrieved metric docs.
- Add evaluation tests that compare both ordered and unordered table outputs, with the comparison mode declared per question.
